# v3 采集优化方案

基于 260K ASIN 采集日志分析（3 个 Worker，17 小时）

## 问题 1：parse_error 占 69%（48,090 次）

**现象**：大量"标题为空"导致 parse_error
**根因**：响应 200 但返回了空壳/降级页面，标题提取为空
**解决方案**：
- Worker 端在解析后、提交前做 `_is_parse_failure` 预检
- 如果标题为空但 HTTP 200，触发 session 轮换（而非仅重试）
- 重试时用新 session，避免被标记的 session 反复拿到降级页面

## 问题 2：代理 429 限流（15,662 次）

**现象**：代理返回 429 Too Many Requests
**根因**：3 个 Worker × 并发 20+ = 总 QPS 60+，超出代理承载
**解决方案**：
- 降低 GLOBAL_MAX_QPS（1000→合理值）
- 或调整 TOKEN_BUCKET_RATE 限制单 Worker QPS
- 监控代理 429 频率，自动降速

## 问题 3：请求超时 21,269 次

**现象**：15s 超时，占总 ERROR 30%
**根因**：代理拥堵时延迟飙升
**解决方案**：
- 动态超时：正常 15s，429 后临时提高到 25s
- 或减少并发降低代理负载

## 问题 4：AIMD 并发波动大

**现象**：16→26→18 来回震荡
**根因**：代理 429 导致成功率下降 → 减速 → 恢复 → 再涨
**解决方案**：
- 增大 ADJUST_INTERVAL（3s→5s）减少调整频率
- 或设置更保守的增速（每次 +1 而非 +2）
- 冷却时间加长避免反弹

## 问题 5：Session 初始化失败 299 次

**现象**：代理不稳定时 session 创建失败
**解决方案**：
- 初始化重试间隔指数退避
- 失败时等待代理恢复而非立即重试

## 优先级排序

1. **[高]** parse_error 预检 + session 轮换 — 直接减少 69% 的错误
2. **[高]** 代理 QPS 限制 — 减少 429 和超时
3. **[中]** AIMD 参数调优 — 减少波动
4. **[低]** Session 初始化容错 — 影响小
